# AGENTS.md — AI 에이전트 운영 가이드

## 개요
이 프로젝트는 하네스 엔지니어링 기반으로,
AI 에이전트가 코드 생성, 스테이지 제작, 테스트, 문서 업데이트를
자율적으로 수행합니다.
사람은 방향을 제시하고, 에이전트는 구현을 담당합니다.

---

## 에이전트 역할 정의

### 1. 🏗️ Implementor (구현 에이전트)
**담당**: Dart/Flame 코드 생성 및 수정

| 작업 | 설명 |
|------|------|
| 컴포넌트 구현 | Ball, Paddle, Brick, ItemDrop 컴포넌트 |
| 시스템 구현 | 충돌 처리, 아이템 효과, 스테이지 로더 |
| UI 구현 | 로비, 가챠, 컬렉션, 튜토리얼 화면 |
| 버그 수정 | 테스트 실패 또는 리포트된 버그 자율 수정 |

**자율 결정 가능한 것:**
- 코드 구조 및 파일 분리 방식
- Flame API 선택
- 성능 최적화 방법 (오브젝트 풀링 등)
- 리팩토링 범위

**사람 승인 없이 하면 안 되는 것:**
- core-beliefs.md의 원칙을 위반하는 기능 추가
- 외부 패키지 신규 추가 (pubspec.yaml 변경)
- 수익화 구조 변경

---

### 2. 🗺️ Stage Builder (스테이지 에이전트)
**담당**: 스테이지 JSON 파일 자동 생성 및 밸런싱

| 작업 | 설명 |
|------|------|
| 스테이지 생성 | stage_001.json ~ stage_010.json 생성 |
| 난이도 커브 적용 | stage-system.md의 구간별 규칙 준수 |
| 아이템 드롭 배치 | 구간별 드롭률 기준으로 벽돌 배치 |
| 패턴 디자인 | 테마에 맞는 벽돌 배치 모양 생성 |

**자율 결정 가능한 것:**
- 벽돌 배치 패턴 및 모양
- HP 분포 (구간 기준 내에서)
- 아이템 드롭 벽돌 위치
- 스테이지 내 빈 공간 구성

**스테이지 생성 시 반드시 지키는 규칙:**
```
1. grid_cols: 7, grid_rows: 10 고정
2. 구간별 HP 범위 준수 (stage-system.md 참고)
3. 구간별 아이템 드롭 비율 준수
4. 1판 목표 클리어 시간 90초~3분 (core-beliefs.md 참고)
5. 판타지 테마 일관성 유지
```

---

### 3. 🧪 QA Agent (테스트 에이전트)
**담당**: 테스트 코드 작성 및 실행

| 작업 | 설명 |
|------|------|
| 단위 테스트 | 충돌 로직, 아이템 효과, 저장소 읽기/쓰기 |
| 스테이지 검증 | JSON 스키마 유효성, 클리어 가능 여부 |
| 회귀 테스트 | 버그 수정 후 기존 기능 정상 작동 확인 |
| 성능 테스트 | 60fps 유지 여부 |

**테스트 통과 기준:**
```
- flutter analyze: 경고 0개
- flutter test: 전체 통과
- 스테이지 JSON 스키마 검증: 전체 통과
- 60fps 유지: 멀티볼 활성화 상태에서도 유지
```

---

### 4. 📝 Doc Agent (문서 에이전트)
**담당**: 문서 자동 업데이트 및 동기화

| 작업 | 설명 |
|------|------|
| 스펙 반영 | 구현 완료된 기능을 product-specs에 반영 |
| 미결 항목 업데이트 | 결정된 항목은 체크박스 체크 처리 |
| 변경 이력 기록 | 주요 결정사항 docs/exec-plans에 기록 |

**문서 업데이트 트리거:**
```
- 새 기능 구현 완료 시 → 관련 product-specs 업데이트
- 미결 항목 결정 시 → 해당 문서 체크박스 처리
- 버그 수정 완료 시 → tech-debt-tracker.md 업데이트
```

## 에이전트 공통 원칙

### 반드시 참조하는 문서
```
1. CONSTITUTION.md                    ← 최상위 원칙 (최우선)
2. docs/design-docs/core-beliefs.md   ← 게임 설계 철학
3. docs/product-specs/index.md        ← 전체 스펙 개요
4. ARCHITECTURE.md                    ← 기술 스택 및 구조
```

### 작업 순서 원칙
```
1. 관련 스펙 문서 확인
2. core-beliefs.md 체크리스트 검토
3. 구현 / 생성 / 테스트 / 문서화
4. 완료 후 관련 문서 업데이트
```

### 에이전트가 절대 하지 않는 것

`CONSTITUTION.md` 5조 불변 원칙 및 `CLAUDE.md` 금지 항목을 따른다.
이 문서에 별도로 중복 기재하지 않는다.

### 미결 의사결정(`open-decisions.md`) 항목을 만났을 때

에이전트 재량으로 판단한다. 판단 기준:

```
맥락상 명확한 경우  → 진행, 완료 후 보고
맥락상 불명확한 경우 → 즉시 멈추고 사람에게 질문
```

"명확하다"의 기준:
- 다른 문서(core-beliefs.md, product-specs 등)에서 방향을 추론할 수 있다
- 보수적인 선택지가 하나로 수렴한다

"불명확하다"의 기준:
- 두 선택지가 비슷하게 합리적으로 보인다
- 결정이 다른 시스템(다른 에이전트의 작업, 타 문서)에 영향을 준다
- 🔴 Blocker 항목이다

### 크로스 에이전트 작업 주도권

여러 에이전트가 동시에 영향받는 작업의 주도권:

| 작업 | 주도 에이전트 | 후속 에이전트 |
|------|------------|------------|
| 새 아이템 타입 추가 | Implementor (코드 + stage_loader 먼저) | Stage Builder (JSON 업데이트) |
| 스테이지 밸런스 조정 | Stage Builder | QA (재검증) |
| 버그 수정 | Implementor | QA (회귀 테스트) |
| 스펙 변경 | 사람 결정 후 | Doc Agent → Implementor 순서 |

## 에이전트 협업 흐름

```
사람: 요청
  ↓
Implementor: 코드 구현
  ↓ [인수인계 조건 충족 시 넘김]
QA Agent: 테스트 작성 및 실행
  ↓ [인수인계 조건 충족 시 넘김]
Stage Builder: 관련 스테이지 JSON 업데이트 (필요 시)
  ↓ [인수인계 조건 충족 시 넘김]
Doc Agent: 문서 동기화
  ↓ [완료 조건 충족 시 선언]
GitHub Actions: 빌드 / 테스트 자동 실행
  ↓
완료
```

### 인수인계 조건

**Implementor → QA**
- `flutter analyze` 경고 0개
- 구현한 기능 목록과 변경된 파일 목록을 명시해서 전달

**QA → Stage Builder**
- `flutter test` 전체 통과
- 핵심 4개 파일(`ball`, `paddle`, `collision_system`, `item_system`) 각각 테스트 1개 이상 존재
- 새로 추가된 UI overlay/HUD가 있을 경우: **상태 반응성 테스트** 1개 이상 존재
  - 검증 기준: 게임 상태(`ValueNotifier` 등) 변경 시 위젯이 올바르게 리빌드되는가
  - 근거: `QUALITY_SCORE.md` 테스트 유형 분류 참고

**Stage Builder → Doc Agent**
- `/run-qa` 스테이지 JSON 검증 전체 통과

**Doc Agent 완료 선언 조건**
- 변경된 기능과 관련된 `product-specs/*.md` 업데이트 완료
- `PLANS.md` 해당 Phase 체크리스트 반영 완료
- `open-decisions.md` 결정된 항목 상태 업데이트 완료

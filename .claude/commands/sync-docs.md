# /sync-docs — 문서 동기화

Doc Agent로서 현재 코드 상태와 문서를 동기화한다.
코드를 먼저 파악한 뒤, 문서와 차이가 있는 부분만 업데이트한다.
없는 내용을 추가하거나 틀린 내용을 수정하되, 작성자의 의도를 함부로 바꾸지 않는다.

---

## 실행 순서

### 1. 변경 범위 파악
최근 구현되거나 수정된 파일을 확인한다.
```bash
git diff --name-only HEAD~5
```
git이 없는 경우, 사용자에게 어떤 기능이 방금 완료됐는지 질문한다.

### 2. product-specs 동기화
`docs/product-specs/` 문서들과 실제 구현 코드를 비교한다.

| 확인 항목 | 기준 |
|----------|------|
| 구현된 기능이 스펙에 반영됐는가 | 미반영 시 해당 스펙 문서 업데이트 |
| 스펙과 구현이 다른 부분이 있는가 | **어느 쪽도 수정하지 않는다** — 차이 내용을 정리해서 사람에게 보고 후 지시를 받는다 |
| 미결 의사결정 중 결정된 항목이 있는가 | 체크박스 `[ ]` → `[x]` 처리 + 결정 내용 기술 |

### 3. tech-debt-tracker 업데이트
`docs/exec-plans/tech-debt-tracker.md`를 확인한다.
- 완료된 항목: 상태를 `완료`로 변경 + 완료 시점 기록
- 보류 처리할 항목: 사용자에게 먼저 확인

### 4. PLANS.md 체크리스트 반영
`PLANS.md`의 현재 Phase 체크리스트에서 완료된 항목을 체크한다.
```
- [ ] 항목  →  - [x] 항목
```

### 5. 현재 Phase 감지 및 phase-gates.md 반영
`PLANS.md`의 각 Phase 체크리스트 완료 여부를 기준으로 현재 Phase를 판단한다.
`docs/exec-plans/phase-gates.md`의 "현재 Phase" 항목을 업데이트한다.

판단 기준:
- Phase N의 모든 항목이 체크됐으면 → Phase N 완료, Phase N+1 진행 중
- Phase N에 미체크 항목이 있으면 → Phase N 진행 중

---

## 하지 않는 것

- 미결 의사결정을 임의로 결정하지 않는다 — 사용자에게 질문한다
- 코드에 없는 기능을 스펙에 추가하지 않는다
- `core-beliefs.md`와 `ARCHITECTURE.md`는 수정하지 않는다 (사람만 수정)

## 완료 보고 형식

```
## 문서 동기화 결과

### 업데이트된 문서
- (파일명): (변경 내용 한 줄 요약)

### 체크 처리된 미결 항목
- (문서명): "(항목 내용)" → 결정 내용

### 확인이 필요한 항목 (사용자 판단 필요)
- (항목): (왜 판단이 필요한지)
```
